# SCRUM.md — процес розробки AI Keeper Director

> Методологія: **Scrum з короткими ітераціями**.
> **Спринт = один робочий день.** Жодних двотижневих циклів.
> Один спринт → демо → ретро → наступний спринт.

---

## 1. Ролі (спрощено для соло/малої команди)

- **Product Owner** — користувач (визначає пріоритети Product Backlog).
- **Developer** — асистент Base44 (бере задачі зі Sprint Backlog, доводить до DoD).
- **Scrum Master** — спільна відповідальність (стежимо за процесом і ретро).

## 2. Цикл одного спринту (1 день)

1. **Sprint Planning** — обрати зі Sprint Backlog задачі на сьогодні (нижче).
2. **Робота** — інкремент має лишати застосунок робочим у будь-який момент.
3. **Sprint Review (Demo)** — що готове й видно в превʼю.
4. **Sprint Retro** — що зайшло / що покращити наступного дня.

## 3. Definition of Done (DoD)

Задача вважається **Done**, лише якщо:

- [ ] Збірка без помилок; усі імпорти резолвляться.
- [ ] Сценарій користувача проходиться від початку до кінця в превʼю.
- [ ] Аудіо: тригер реально грає і реально зупиняється (без «зависань»).
- [ ] Нова поведінка не ламає наявну (Play-режим, гучність, обране).
- [ ] Зміни відображені в `TASKS.md` (перенесено в Done) і, за потреби, в `DECISIONS.md`.
- [ ] Жодних заглушок / напівробочих кнопок.

## 4. Артефакти

- **Product Backlog** → `ROADMAP.md` (розділ «Product Backlog»). Великі цілі, пріоритезовані.
- **Sprint Backlog** → `TASKS.md`. Задачі поточного/найближчого спринту.
- **Sprint Review + Retro** → `TASKS.md` (розділ «Журнал спринтів»). Короткий запис на спринт.
- **Definition of Done** → цей файл, розділ 3.

## 5. Правила пріоритезації Product Backlog

1. **Надійність аудіо** важливіша за нові фічі (директор не може «зависнути» в сесії).
2. **Sound > новий екран** — спершу робить наявне бездоганним.
3. Кожен пункт беклогу формулюється як цінність для ведучого (Game Master), а не як «технічна задача».
